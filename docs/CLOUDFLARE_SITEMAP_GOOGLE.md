# Google & sitemap.xml — Cloudflare check and fix

If Google Search Console shows **"Sitemap could not be read"** or **"Couldn't fetch"** for `https://extensionshield.com/sitemap.xml`, the request is often being **challenged or blocked by Cloudflare** (WAF, Bot Fight Mode, or security rules). Browsers may still load the URL because they send different headers and pass challenges.

---

## 1. Check if the sitemap is blocked

### From your machine (simulate Googlebot)

```bash
# As a normal request (might get 403/429 if Cloudflare blocks bots)
curl -i https://extensionshield.com/sitemap.xml

# As Googlebot (what Google uses)
curl -i -A "Googlebot/2.1 (+http://www.google.com/bot.html)" https://extensionshield.com/sitemap.xml
```

- **200 OK** and valid XML → sitemap is reachable; if GSC still fails, use [URL Inspection](https://search.google.com/search-console) and "Test live URL" / "Request indexing".
- **403, 429, or challenge page** → Cloudflare is blocking or challenging the bot. Apply the fix below.

### In Google Search Console

1. Open [Google Search Console](https://search.google.com/search-console) → **Sitemaps**.
2. If the sitemap shows an error like "Couldn't fetch" or "Sitemap could not be read", Cloudflare is likely blocking Googlebot.

---

## 2. Fix: Allow verified bots in Cloudflare for sitemap and robots

So that **Googlebot and other verified crawlers** can read `/sitemap.xml` and `/robots.txt` without being challenged:

### Use the domain (zone), not Account-level WAF

- **Do not** use **Account-level WAF** (that’s an Enterprise add-on and shows “Purchase add-on”).
- **Do** open the **domain/zone** for your site (e.g. you’re on **Overview** for extensionshield.com).
- **From the zone’s left sidebar:** click **Security** → **Security rules**. On that page, look for **Custom rules** (or a “WAF” / “Custom rules” tab) → **Create rule**.  
  *(If your sidebar shows “WAF” under Security, use **Security** → **WAF** → **Custom rules** instead.)*

### Create the rule (zone-level)

1. **Rule name:** e.g. `Allow verified bots for sitemap and robots`
2. **Expression (Edit expression):**
   ```text
   (http.request.uri.path eq "/sitemap.xml" or http.request.uri.path eq "/robots.txt") and cf.client.bot
   ```
3. **Action:** **Skip** → enable **All remaining custom rules** (and **Rate limiting** if you use it for these paths).
4. **Deploy** the rule.

**Result:** Verified bots (e.g. Googlebot) can fetch `/sitemap.xml` and `/robots.txt` without being challenged; other traffic is unchanged.

### No “Custom rules” under Security → WAF?

If your **domain’s** Security → WAF page doesn’t show **Custom rules** (e.g. your plan doesn’t include it):

- **Option A — Config Rules:** In the left sidebar for the domain, check **Rules** → **Config Rules**. If available, add a rule that **skips** or **allows** when:
  - URI path is `/sitemap.xml` or `/robots.txt`, and
  - the request is from a verified bot (if the rule builder exposes something like “Verified Bot” or “cf.client.bot”).
- **Option B — Bots:** Go to **Security** → **Bots**. If **Bot Fight Mode** or **Super Bot Fight Mode** is on, see if you can **allowlist verified bots** (Googlebot, Bingbot) so they are not challenged. That can reduce blocks on sitemap/robots without WAF.
- **Option C — Test first:** Run the curl from section 1. If you get **200 OK** as Googlebot, Google may already be able to read the sitemap; the GSC error might be temporary. Resubmit the sitemap in GSC and recheck.

---

## 3. Optional: Bot Fight Mode / Super Bot Fight Mode

If you have **Bot Fight Mode** or **Super Bot Fight Mode** enabled (Security → Bots):

- They can still challenge or block bot-like requests before your custom rule runs, depending on order of execution.
- Either:
  - Rely on the WAF custom rule above and ensure it runs **before** bot mode (e.g. put "Allow verified bots for sitemap and robots" in a high-priority position), or
  - In **Security** → **Bots**, configure to **allow verified bots** (Googlebot, Bingbot, etc.) so they are not challenged.

---

## 4. Verify after the fix

1. Wait 1–2 minutes for the Cloudflare rule to propagate.
2. Run again:
   ```bash
   curl -i -A "Googlebot/2.1 (+http://www.google.com/bot.html)" https://extensionshield.com/sitemap.xml
   ```
   Expect **200 OK** and valid XML.
3. In GSC → **Sitemaps**, use **Resubmit** (or remove and re-add the sitemap URL). After the next crawl, errors should clear if the only issue was Cloudflare blocking.
